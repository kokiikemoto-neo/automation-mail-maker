# Writing Robot 営業ツール — プロジェクト全体設計

手書きDMによる法人営業を自動化するためのプロジェクト。
会社を選ぶと、情報を収集し、刺さる文面を設計し、ロボットが手書きで便箋を書き起こす。

このドキュメントは「全体の地図」です。各フェーズが進むごとに、
詳細は `docs/` 以下の個別ドキュメントに切り出していきます。

---

## 1. 最終ゴール

```
会社を選択
   ↓
その会社の情報を自動収集（HP・ニュース・業界情報）
   ↓
AI が商談設計・ニーズ分析 → 背中を押すメッセージを生成
   ↓
手書き便箋のレイアウト設計（文字数・行数・配置）
   ↓
G コード生成
   ↓
ロボットが自動で手書き
```

- 利用者：チーム5人が各自で使う
- ターゲット：中小企業を幅広く（数を打つ）
- 想定量：1日 10〜50通
- 用途：法人営業の手書きDM（ネオキャリアの営業ツール）

---

## 2. 全体アーキテクチャ

システムは「クラウド側」と「ローカル側」の2層に分ける。
理由：5人がどこからでも文面を作れる一方、ロボットは物理的に1台しかないため。

```
┌─ クラウド側（誰でも・どこでも）─────────────────┐
│  ①会社選択 → ②情報収集 → ③文面生成 → ④承認        │
│                              ↓                    │
│                      承認済みデータ（キュー）       │
└──────────────────────┬──────────────────────────┘
                       ↓
┌─ ローカル側（ロボットがある場所の専用PC 1台）──────┐
│  ⑤Gコード生成 → ⑥機械へ送信 → ⑦手書き出力         │
└──────────────────────────────────────────────────┘
```

### 2層に分ける効果

- 5人は各自のブラウザで「文面を作って承認」までを行う
- 実際に紙に書く工程だけが、ロボット接続済みの専用PC 1台に集約される
- 間に「書き出し待ちキュー」を挟むことで、5人が好きなタイミングで仕込み、
  専用PCが順番に処理する（差し込み印刷の「印刷ボタン」と「プリンタ」の関係）

### 専用PCについて

- 新規購入は不要。ロボット接続に成功した既存PCを「ロボット担当」に固定する
- このPC上で常駐プログラムが動き、キューを数秒ごとに監視
- 新しい仕事が入ったら G コードに変換して送信
- 5人はこのPCの存在を意識しない

## 3. データの置き場所（重要な設計判断）

商談情報の「本体DB」は未定（これから決める）。
ただし、間に「キュー」を1枚挟むことで、本体DBの種類を専用PCから隠蔽する。

```
商談情報の本体          文面生成        キュー         専用PC
（後で決める）   →      ＋承認    →   書き出し待ち  →  ここだけ見る
  ・スプレッドシート
  ・Salesforce
```

### この分離の効果

- 本体DBがスプレッドシートでも Salesforce でも、変わるのは「本体DBを読む部分」だけ
- 専用PCのプログラムは常に同じ形式のキューを見るだけなので、影響を受けない

### 当面の方針

- 最初はスプレッドシートで始めるのを推奨（導入コストほぼゼロ・5人がすぐ使える）
- Salesforce へは、商談管理を本格的にそこへ寄せると決めた時点で乗り換え
- キュー分離のおかげで、乗り換え時も専用PC側は無改修

## 4. 実装アプローチ（段階的に育てる）

| 案 | 構成 | 手間 | 5人共有 | 期間 |
|---|---|---|---|---|
| A：シート方式 | スプレッドシート＋Pythonスクリプト | 小 | △ | 数日 |
| B：軽量ツール | 簡易Web画面＋共有DB | 中 | ◎ | 1〜2週間 |
| C：フルアプリ | Next.js＋FastAPI | 大 | ◎ | 1ヶ月＋保守 |

### 進め方

**案A で書字品質を確認 → 案B に育てる。**

理由：このプロジェクトの最大の不確実性は「ロボットがきれいに字を書けるか」。
文面生成や共有UIは後から作り込めるが、書字品質（かすれ・歪み・速度）は
実際に書かせてみないとわからない。ここが土台なので最初に潰す。

案C（フルアプリ）は、外部企業へのSaaS外販を本気で狙う段階になってから検討。
5人・社内利用の現規模には過剰。

いきなり漢字を書かせず、段階を踏む。

- [x] ステップ1：直線テスト（横線・縦線）`python test_write.py 1` ← 完了（横線・縦線OK）
- [x] ステップ2：四角形テスト（角が閉じるか）`python test_write.py 2` ← 完了（正方形OK）
- [x] ステップ3：ひらがな「し」（曲線がなめらかか）`python test_write.py 3` ← 完了（ベジェ多点でなめらか）
- [x] ペン上下の制御調整 → Z軸方式・向き逆・G4同期で確定
- [x] 速度（FEED）：1000 で良好

> 学び：曲線のなめらかさは「機械精度」ではなく「点の細かさ」で決まる。
> 8点だと折れ線に見え、24点に増やすとなめらか。Phase 2 の SVG変換で自動分割される。

> ここが NG なら、ペン・機械側の調整が先。
> ここが OK なら Phase 2 へ。

### Phase 2：文字を書く仕組み（フォント対応） ✅ ほぼ完了

目標：任意の日本語テキストを、単線で書けるようにする。

> 実装メモ：当初想定の「SVG経由(vpype)」ではなく、フォント輪郭(freetype)と
> KanjiVG筆画を直接Gコード化する方式に変更。輪郭フォントは二重線（袋文字）に
> なるため、単線にするには KanjiVG（筆順ストロークの中心線）が必要だった。

- [x] 英数字の単線（Hershey フォント）`text2gcode.py` — 「NEO」成功
- [x] 日本語の単線（KanjiVG 筆画）`kanji2gcode.py` — 「池本光輝」成功（単線・筆順どおり）
- [x] 位置決めと座標系の確立（--frame / --origin / --paper）
- [x] 全 KanjiVG データ導入（11,662文字。任意の漢字・かな・カナ対応）
- [x] 縦書き・複数行レイアウト（`--vertical` / `\n` 改行）
- [x] 宛名の標準文字高さ＝**5mm** に確定
- [x] 速度改善（ペン上下6倍・描画2.5倍・同期待ち半減）
- [x] 本文（数行）の縦書き・段差つきレイアウト
- [x] 手書きらしい“ゆらぎ”の付加（フィボナッチmod3段差・文字位置ゆらぎ）

### 書字エンジン 確定設定（2026-06 時点）

メインは `skeleton2gcode.py`（輪郭フォントを単線化して書く）。最終的に到達した設定：

| 項目 | 確定値・内容 |
|---|---|
| フォント | **ひとりゴシック**（`fonts/HitoriGothic.ttf`、約7000字・手書き風・商用可） |
| 線 | **単線化**（skeletonize で中心線抽出。二重線にならない） |
| ペン下げ Z | **10.0**（紙に着く。向きは逆＝Z大で下がる） |
| ペン上げ Z | **7.0**（＝上げ幅3mm。これ未満だと移動中に擦る。実測で確定） |
| 速度 | `PEN_FEED=10000` / `FEED=6000` / 同期 `G4 P0.03` |
| 文字サイズ | 漢字/かな別指定（例 漢字8mm・かな6mm）。`--kana` |
| 接頭辞の例外 | 「お」「ご」の次が漢字なら漢字サイズ（お慶び・ご清栄） |
| 字形比 | `GLYPH_X=0.90` / `GLYPH_Y=1.12`（少し縦長） |
| 段差 | **フィボナッチmod3**（0,1,1,2,0,2,2,1,0…）× 半角（`head_step=漢字h×0.5`） |
| 手書き感 | 文字位置ゆらぎ（たまに `BIG_JIT` で大きくずれる） |
| 縦書き | `--vertical`（上→下、改行で右→左の列） |
| USB対策 | **分割送信**（`draw_chunked`、40本ごとに再接続。切断に強い） |
| 原点 | 電源ON時のヘッド位置が原点。書く前に紙の左下へヘッドを置く |

> 既知の課題：USB（CH340）接続が不安定で、長文の連続書き込み中に切断/デバイス消失が起きる。
> 分割送信＋再接続で粘るが、根本対策はUSBケーブル・ポート・電源の安定化（ハード側）。

## 7. 確定済みの機械情報（Phase 0 の成果）

| 項目 | 値 | 意味 |
|---|---|---|
| 機種 | Writing Robot T-A4 (steamduino) | XY プロッター型 |
| ファームウェア | GRBL 1.1h | CNC 制御 |
| 接続 | USB（COM3）/ 115200 bps | |
| X 最大移動 | 210mm | A4 横幅 |
| Y 最大移動 | 297mm | A4 縦幅 |
| Z 最大移動 | 10mm | ペンの上下 |

### ペン上下制御（Phase 1 の実機検証で判明・重要）

- ペンの上下は **Z軸**（`G0/G1 Z`）で制御する。**サーボ（M3/M5）ではない**
- **向きが一般と逆**：`Z=10` で紙に着く（ペン下げ）／`Z=0` で離れる（ペン上げ）
- 各ペン上下の直後に `G4 P0.3`（ドウェル）で**同期**を取る。
  これが無いと Z移動が終わる前に XY が動き出し、ペンが降りきらず線がかすれる／書けない
- ファーム設定メモ：`$30=1000`（PWM最大）, `$32`（レーザーモード）。
  ペンはZ軸なので `$32` はペン制御に無関係

> これらは `tests/test_write.py`（現状はルート直下）の冒頭設定に反映済み。

### 座標系と原点（Phase 2 で判明・重要）

- **原点(0,0)は電源ON時のヘッド位置**で決まる（絶対原点を持たない）。
  運用：電源OFF→ヘッドを手で紙の左下角へ→電源ON、で「紙の左下=原点」にする。
- **X+ = 紙の右、Y+ = 紙の上**
- Y を大きくしすぎると機械上限に当たり脱調する（ゴリゴリ鳴る）。原点位置しだいで
  上限が変わるので、`--frame`（枠なぞり）で位置を確認してから書く。
- 紙サイズは毎回変わる（最終的に便箋）。`--paper W H` で都度指定してはみ出しチェック。

## 8. リポジトリ構成（推奨）

プロジェクトが育つにつれ、こう整理していくと見通しが良い：

```
writing-robot/
├── README.md            ← このプロジェクトの入口（概要とリンク）
├── ROADMAP.md           ← このファイル（全体設計・進捗）
├── SETUP.md             ← 環境構築手順
├── docs/                ← フェーズごとの詳細設計
│   ├── phase1-quality.md
│   ├── phase2-fonts.md
│   └── ...
├── src/                 ← 本番コード
│   ├── robot/           ← ロボット制御（Gコード送信）
│   ├── text2gcode/      ← テキスト→SVG→Gコード
│   └── queue/           ← キュー監視・専用PC常駐
├── tests/               ← テスト用スクリプト
│   └── test_write.py    ← 書字品質テスト（Phase 1）
└── data/                ← サンプルの宛名・文面（CSV など）
```

> 最初から全部作る必要はない。Phase が進むごとに必要なフォルダを足していく。
> いまは `tests/test_write.py` と各 `.md` があれば十分。
