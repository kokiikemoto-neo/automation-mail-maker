# Writing Robot 営業ツール — プロジェクト全体設計

手書きDMによる法人営業を自動化するためのプロジェクト。
会社を選ぶと、情報を収集し、刺さる文面を設計し、ロボットが手書きで便箋を書き起こす。

このドキュメントは「全体の地図」です。各フェーズが進むごとに、
詳細は `docs/` 以下の個別ドキュメントに切り出していきます。

---

## 1. 最終ゴール

```
会社を選択
   ↓
その会社の情報を自動収集（HP・ニュース・業界情報）
   ↓
AI が商談設計・ニーズ分析 → 背中を押すメッセージを生成
   ↓
手書き便箋のレイアウト設計（文字数・行数・配置）
   ↓
G コード生成
   ↓
ロボットが自動で手書き
```

- 利用者：チーム5人が各自で使う
- ターゲット：中小企業を幅広く（数を打つ）
- 想定量：1日 10〜50通
- 用途：法人営業の手書きDM（ネオキャリアの営業ツール）

---

## 2. 全体アーキテクチャ

システムは「クラウド側」と「ローカル側」の2層に分ける。
理由：5人がどこからでも文面を作れる一方、ロボットは物理的に1台しかないため。

```
┌─ クラウド側（誰でも・どこでも）─────────────────┐
│  ①会社選択 → ②情報収集 → ③文面生成 → ④承認        │
│                              ↓                    │
│                      承認済みデータ（キュー）       │
└──────────────────────┬──────────────────────────┘
                       ↓
┌─ ローカル側（ロボットがある場所の専用PC 1台）──────┐
│  ⑤Gコード生成 → ⑥機械へ送信 → ⑦手書き出力         │
└──────────────────────────────────────────────────┘
```

### 2層に分ける効果

- 5人は各自のブラウザで「文面を作って承認」までを行う
- 実際に紙に書く工程だけが、ロボット接続済みの専用PC 1台に集約される
- 間に「書き出し待ちキュー」を挟むことで、5人が好きなタイミングで仕込み、
  専用PCが順番に処理する（差し込み印刷の「印刷ボタン」と「プリンタ」の関係）

### 専用PCについて

- 新規購入は不要。ロボット接続に成功した既存PCを「ロボット担当」に固定する
- このPC上で常駐プログラムが動き、キューを数秒ごとに監視
- 新しい仕事が入ったら G コードに変換して送信
- 5人はこのPCの存在を意識しない

---

## 3. データの置き場所（重要な設計判断）

商談情報の「本体DB」は未定（これから決める）。
ただし、間に「キュー」を1枚挟むことで、本体DBの種類を専用PCから隠蔽する。

```
商談情報の本体          文面生成        キュー         専用PC
（後で決める）   →      ＋承認    →   書き出し待ち  →  ここだけ見る
  ・スプレッドシート
  ・Salesforce
```

### この分離の効果

- 本体DBがスプレッドシートでも Salesforce でも、変わるのは「本体DBを読む部分」だけ
- 専用PCのプログラムは常に同じ形式のキューを見るだけなので、影響を受けない

### 当面の方針

- 最初はスプレッドシートで始めるのを推奨（導入コストほぼゼロ・5人がすぐ使える）
- Salesforce へは、商談管理を本格的にそこへ寄せると決めた時点で乗り換え
- キュー分離のおかげで、乗り換え時も専用PC側は無改修

## 4. 実装アプローチ（段階的に育てる）

| 案 | 構成 | 手間 | 5人共有 | 期間 |
|---|---|---|---|---|
| A：シート方式 | スプレッドシート＋Pythonスクリプト | 小 | △ | 数日 |
| B：軽量ツール | 簡易Web画面＋共有DB | 中 | ◎ | 1〜2週間 |
| C：フルアプリ | Next.js＋FastAPI | 大 | ◎ | 1ヶ月＋保守 |

### 進め方

**案A で書字品質を確認 → 案B に育てる。**

理由：このプロジェクトの最大の不確実性は「ロボットがきれいに字を書けるか」。
文面生成や共有UIは後から作り込めるが、書字品質（かすれ・歪み・速度）は
実際に書かせてみないとわからない。ここが土台なので最初に潰す。

案C（フルアプリ）は、外部企業へのSaaS外販を本気で狙う段階になってから検討。
5人・社内利用の現規模には過剰。

## 5. 開発フェーズ（ロードマップ）

### Phase 0：ロボット制御の基盤 ✅ 完了

- [x] ロボットとPCをUSB接続
- [x] GRBL 通信確認（GRBL 1.1h）
- [x] UGS で接続・設定値確認（X最大210mm / Y最大297mm = A4対応）
- [x] ヘッドが実際に動くことを確認

### Phase 1：書字品質の検証 ◀ いまここ

目標：日本語の宛名と短い本文を1枚、実際に書かせて品質を見る。
いきなり漢字を書かせず、段階を踏む。

- [ ] ステップ1：直線テスト（横線・縦線）`python test_write.py 1`
- [ ] ステップ2：四角形テスト（角が閉じるか）`python test_write.py 2`
- [ ] ステップ3：ひらがな「し」（曲線がなめらかか）`python test_write.py 3`
- [ ] ペン上下の制御調整（必要なら）
- [ ] 速度（FEED）の最適値を見つける
- [ ] 判定：書字品質は営業DMとして許容できるか？

> ここが NG なら、ペン・機械側の調整が先。
> ここが OK なら Phase 2 へ。

### Phase 2：文字を書く仕組み（フォント対応）

目標：任意の日本語テキストを、手書き風フォントで書けるようにする。

- [ ] テキスト → SVG 変換（手書き風フォント適用）
- [ ] SVG → G コード変換（vpype などを利用）
- [ ] 宛名フルセット（会社名・部署・氏名）を1枚書く
- [ ] 本文（数行のメッセージ）を書く
- [ ] レイアウト設計（便箋の余白・行間・文字サイズ）
- [ ] 手書き風フォントの選定 or 自作（自分の字の取り込みも可能）

### Phase 3：差し込み自動化（案A の完成形）

目標：スプレッドシートの宛名・メッセージ一覧から、1人ずつ自動で書く。

- [ ] スプレッドシート（CSV）から宛名・本文を読み込む
- [ ] 1件ずつ「SVG生成 → Gコード変換 → 送信」をループ
- [ ] 1枚書き終わったら次へ進む制御（紙交換の合図など）
- [ ] エラー時の停止・再開

> この時点で「シートに書けば順番に手書きされる」状態。案A 完成。

### Phase 4：文面の自動生成（AI 連携）

目標：会社名を入れると、刺さる文面が自動で出てくる。

- [ ] Claude API で文面生成（会社情報を入力 → メッセージ出力）
- [ ] ニーズ分析のプロンプト設計（採用課題・離職・人手不足など）
- [ ] 文面のテンプレ感を避ける工夫（個別性を持たせる）
- [ ] 生成 → 人が承認するフロー
- [ ] 承認済みのものだけキューに入る

### Phase 5：情報収集の自動化

目標：会社名を入れると、その会社の情報を自動で集める。

- [ ] Web 検索連携（HP・ニュース・業界情報）
- [ ] 収集情報の構造化（事業内容・規模・最近の動き）
- [ ] 文面生成への受け渡し

### Phase 6：5人で使える共有ツール（案B）

目標：5人が各自のブラウザから使える形にする。

- [ ] 共有DB（キュー）の構築
- [ ] 簡易Web画面（会社選択・文面確認・承認・進捗表示）
- [ ] 専用PC の常駐プログラム（キュー監視 → 自動書き出し）
- [ ] 誰が・いつ・どの会社に送ったかの記録

### Phase 7（将来）：スケール・外販

- [ ] 複数台のロボット対応
- [ ] 量産運用
- [ ] 外部企業向け SaaS 化の検討（案C）

## 8. リポジトリ構成（推奨）

プロジェクトが育つにつれ、こう整理していくと見通しが良い：

```
writing-robot/
├── README.md            ← このプロジェクトの入口（概要とリンク）
├── ROADMAP.md           ← このファイル（全体設計・進捗）
├── SETUP.md             ← 環境構築手順
├── docs/                ← フェーズごとの詳細設計
│   ├── phase1-quality.md
│   ├── phase2-fonts.md
│   └── ...
├── src/                 ← 本番コード
│   ├── robot/           ← ロボット制御（Gコード送信）
│   ├── text2gcode/      ← テキスト→SVG→Gコード
│   └── queue/           ← キュー監視・専用PC常駐
├── tests/               ← テスト用スクリプト
│   └── test_write.py    ← 書字品質テスト（Phase 1）
└── data/                ← サンプルの宛名・文面（CSV など）
```

> 最初から全部作る必要はない。Phase が進むごとに必要なフォルダを足していく。
> いまは `tests/test_write.py` と各 `.md` があれば十分。

## 9. いますぐの次の一手

1. このファイル（ROADMAP.md）と SETUP.md、test_write.py をリポジトリに置く
2. SETUP.md に沿って環境を整える
3. `python test_write.py 1` で直線テスト
4. 結果（書けた / かすれた / 動かない 等）を確認
5. Phase 1 のチェックリストを埋めていく
